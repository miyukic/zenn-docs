Claude Codeで進めた作業の続きを、翌日にCodexで始める。外出中はスマートフォンのChatGPTから確認する。

この時、それぞれのAIは昨日の会話を自動では知りません。AIごとに会話履歴も保存場所も違うためです。

Locus Systemは、この分断を埋めるために作っている個人用の記憶システムです。

目標は、**会話、作業、判断、失敗を含むすべての記憶を完全な形で残し、必要な瞬間にAIが文脈として思い出せること**です。

最終的には、利用者が裏側のファイルや保存処理を意識する必要もなくします。普通にAIと話し、作業するだけで記憶が残り、別のAIへ移っても自然に続きが始まる状態を目指しています。

この目標はまだ完成していません。

現在は、次の3つを組み合わせて動かしています。

1. 人が読めるMarkdownファイル
2. AIがどのファイルを読み書きするか決めた運用ルール
3. 過去の記録を検索し、変更分を保存するプログラム

つまり、MarkdownはLocus Systemの最終形ではありません。すべての記憶を残して自然に思い出すという目標を、今の技術で実際に動かすための保存方法です。

## なぜ作ったのか

AIと長期プロジェクトを進めると、毎回同じ問題が起きます。

- 新しい会話を始めると、前回の決定を知らない
- AIを替えると、最初から説明し直しになる
- 作業ログが長くなると、すべてを毎回読み込めない
- 専用のプロジェクトメモがあっても、AIが存在を知らなければ読まれない
- AIが提案したことと、本人が決めたことが混ざる

会話を一つだけ使い続ければ一時的には避けられます。しかし、会話が長くなるほど扱いにくくなり、AIやサービスを乗り換えることも難しくなります。

そこで、会話の外に記憶を置くことにしました。

ただし、人が毎回「これはどのファイルへ書くか」と管理し続けることが目的ではありません。現在はAIが決められたルールに従って記録し、将来は保存先そのものを意識しなくてよい状態へ近づけます。

## 現在の実装は一つのフォルダ

現在のLocus Systemでは、一つのフォルダを記憶の中心にしています。

```text
locus/
├─ rules.md
├─ history_summary.md
├─ ai_handoff.md
├─ me.md
├─ log.md
├─ projects/
├─ me/
├─ history/
└─ skills/
```

それぞれの役割を分けています。

| 場所 | 入っているもの |
|---|---|
| `rules.md` | AIが守る操作ルール |
| `history_summary.md` | 初期の出来事を短くまとめた年表。現在は更新停止中 |
| `ai_handoff.md` | 今の作業、残り作業、次のAIに必要な注意 |
| `me.md` | 利用者の好みや、毎回必要になる前提 |
| `log.md` | 日付順に追記する作業記録 |
| `projects/` | プロジェクトごとの詳しい記録 |
| `me/` | 毎回は不要だが、必要な時に読む個人側の詳細 |
| `history/` | 各AIの会話履歴をMarkdownへ変換したもの |
| `skills/` | 食事記録や検索など、複数AIで共有する機能 |

データ形式をMarkdownにした理由は単純です。

- 人がそのまま読める
- AIが特別なソフトなしで読める
- 手作業でも修正できる
- Gitで変更履歴を残せる
- 特定のAI会社やモデルに依存しない

データベースが壊れても、現在の記憶はMarkdownとして残ります。

## AIが起動した時に読むもの

すべての記録を毎回読むと、入力が大きくなりすぎます。

そこで起動時は、必要度の高い情報だけを決まった順番で読みます。

```text
1. rules.md
   操作ルールを確認

2. history_summary.md
   初期の経緯を確認（現在は更新停止中）

3. ai_handoff.md
   現在地と残り作業を確認

4. me.md
   利用者側の基本前提を確認

5. log.mdの末尾50行
   直前の会話や作業を確認
```

`history_summary.md`は、長くなった`log.md`の古い部分を短くまとめるために作りました。しかし実際には2026年4月を最後に要約作業が止まっており、現在は起動時に読まれるものの、新しい出来事は入っていません。

最近の引き継ぎは、主に`ai_handoff.md`、`log.md`の末尾、必要な時のRAG検索が担っています。`history_summary.md`を定期的に更新するか、別の方法へ置き換えるかは、現在残っている運用上の課題です。

これらを読むことで、AIは「このシステムをどう扱うか」「今どこまで進んでいるか」「誰と話しているか」を最初に把握します。

起動時に読むのは、中核ファイルと直近ログだけです。

数か月前の細かい出来事や、特定プロジェクトの詳細は必要になった時だけ検索します。

## 会話中は何を書き分けるのか

一つのファイルに全部書くと、後から探せなくなります。

そのため、内容によって保存先を分けます。

```text
今日何をしたか
  → log.md

プロジェクト固有の決定や検証結果
  → projects/プロジェクト名.md

次のAIが必ず知る必要がある現在地
  → ai_handoff.md

長期間変わらない利用者側の前提
  → me.md または me/配下

環境固有の操作方法や障害対応
  → knowledge.md / env.md
```

`log.md`は追記専用です。古い記録を削除しません。

また、記録へ「決定」と書けるのは、本人が実際に決めたことか、検証で確定したことだけです。AIの提案を、実施済みの事実として保存しないようにしています。

複数AIが同じファイルを書くため、各記録には`[Codex]`や`[CC]`のような著者名も付けます。

## 会話が終わる時に何を保存するのか

会話の区切りでは、次の処理を行います。

```text
今回の作業をlog.mdへ追記
        ↓
ai_handoff.mdと該当プロジェクトを更新
        ↓
各AIの新しい会話履歴をMarkdownへ追加
        ↓
Gitへ保存
        ↓
変更された文章だけ検索DBへ反映
```

この一連の処理をLocusSaveと呼んでいます。

各AIツールが持つ元の会話履歴は、保存形式がばらばらです。対応している履歴は、人とAIの発言を取り出してMarkdownへ変換します。

この時、画像本体や長い機械出力は入れず、秘密情報らしい文字列も伏せます。履歴ファイル全体を毎回作り直すのではなく、前回保存した位置より後ろに増えた部分だけを追加します。

## なぜ全部を直接読まず、検索するのか

`log.md`や`history/`は、使い続けるほど大きくなります。すべてをAIへ渡す方式では、記録が増えるほど起動が遅くなり、入力も増え続けます。

そこで、情報を二つに分けています。

### 毎回そのまま読む

- 操作ルール
- 初期の長期要約（現在は更新停止中）
- 現在の作業
- 利用者側の基本前提
- 直近のログ

### 必要な時だけ検索する

- 過去の詳しい作業記録
- プロジェクトごとの記録
- AIとの古い会話
- 環境や障害対応の記録

たとえば「前に3Dプリンターの寸法補正で何を試したか」と聞かれた時だけ、その話に近い記録を探します。

この分け方により、記録が増えても起動時に読む量をほぼ一定にできます。

## 過去の記録をどう検索しているのか

現在の検索部分は次の構成です。

| 役割 | 使用しているもの |
|---|---|
| Markdownの取り込みと検索 | C# / .NET 9 |
| 文章を検索用の数値へ変換 | Ollama + `bge-m3` |
| 検索用データの保存 | PostgreSQL + pgvector |
| AIからの読み取り窓口 | コマンド、ネットワーク越しの読み取り口、ChatGPTの外部接続方式（MCP） |

RAGは、過去の記録から質問に近い文章を探し、その文章を現在のAIへ渡す部分です。

元のMarkdownを置き換えるものではありません。検索用データベースは、Markdownから作り直せる索引として扱っています。

文章を検索用の数値へ変えたものを、この記事ではベクトルと呼びます。

現在は、本文1行を一つの検索単位にします。ただし、本文だけでは何の話か分からなくなるため、直前の見出しを検索用の文章へ付け足します。

たとえば元ファイルが次の形だったとします。

```markdown
## 3Dプリンターの寸法補正

- 穴が設計値より小さく出た
- 別の材料でも同じ試験を行う
```

検索用には、次のような文章を作ります。

```text
3Dプリンターの寸法補正 / 穴が設計値より小さく出た
3Dプリンターの寸法補正 / 別の材料でも同じ試験を行う
```

検索結果には元ファイル名と行番号も入るため、必要なら前後の文章を読み直せます。

## 消した文章のベクトルが、検索結果に残っていた

現在のRAGは、記録した文章をベクトルへ変換し、検索用のDBへ保存しています。ベクトルは文章の意味を数値で表したもので、質問と意味が近い文章を探すために使います。このDBは、元の記録をすばやく探すための索引にあたります。

この検索用DBの更新に、一つバグがありました。元の記録から文章を消した後も、その文章から作ったベクトルがDBに残り、検索すると修正前の文章が出てくることがありました。

```text
修正前の記録: 穴が設計値より小さく出た
修正後の記録: （この文章はもう存在しない）

検索結果: 穴が設計値より小さく出た  ← 消した文章が出てくる
```

原因は、追加・変更された文章のベクトルはDBへ保存していたのに、消された文章の古いベクトルを十分に片づけていなかったことです。元の記録は正しく直っていても、検索だけが古い内容を覚えていました。

そこで、記録を取り込むたびに、現在の文章とDB内のベクトルを照らし合わせるようにしました。元の記録に対応する文章が存在しないベクトルは削除します。修正後にすべての記録を取り込み直したところ、古いベクトルが631件見つかり、削除されました。

「古いベクトルが残る」とは、元の文章はすでに消えているのに、その文章を検索するための数値だけがDBに残っていた、という意味です。

## 複数のAIはどうつながっているのか

ローカルで動くAIツールは、同じLocusフォルダを直接読み書きします。

スマートフォンやWeb版ChatGPTはローカルファイルを直接読めないため、読み取り専用の窓口を用意しています。

```text
ローカルのAI
  → 共有フォルダと検索コマンド

ChatGPT
  → 外へ向けて張った安全な接続
  → Locus API
  → 同期・検索・前後行の取得
```

ChatGPTからは、現在の作業、利用者側の基本前提、直近ログを一度に取得できます。更新停止中の長期要約も取得できますが、最近の詳しい記録はRAGで検索します。

現時点のChatGPT側は読み取り専用です。ローカルのAIと同じように自由に書き換えるところまでは開放していません。

重要なのは、現在のAI同士が見えない場所で直接通信しているわけではないことです。

それぞれのAIは独立しており、今の実装ではLocusへ保存された記録だけを共有します。保存されていない作業や会話は、別のAIからは見えません。

これはLocus Systemの定義ではなく、現在残っている制限です。目標は、会話や作業を取りこぼさず自動で記録し、別のAIが必要な部分を自分で思い出せる状態です。

## 壊れにくくするためのルール

複数AIが長期間同じ記録を使うため、便利さより先に守る境界を決めています。

- `log.md`は追記専用で、過去を削除しない
- ファイルの削除や上書きは本人確認なしで行わない
- AIの提案を本人の決定として記録しない
- 必須フォルダが見えない時は、検索DBの削除処理を止める
- 文字コードを固定し、日本語の文字化けを防ぐ
- Gitで変更履歴を残す
- 公開記事には、個人情報、LAN内の接続先、秘密情報を出さない

検索DBやAPIは作り直せますが、元のMarkdownと履歴は作り直せません。そのため、守る順番は常に「元ファイルが先、便利な仕組みは後」です。

## 現在できていることと、まだできていないこと

現在できることは次の通りです。

- 別のAIでも、同じ現在地から作業を再開する
- 過去の決定や失敗を、必要な時だけ検索する
- プロジェクトごとの記録を、人とAIの両方が読む
- AIやモデルを乗り換えても、記憶の形式を維持する
- スマートフォンのChatGPTからローカルの記憶を読む

一方で、現在は次のことが完全には自動化できていません。

- すべての会話や作業が、何もしなくても必ず記録される
- AIの内部状態や考えが、そのまま別AIへ伝わる
- 書かれていない出来事を思い出す
- 検索すれば必ず正解が出る
- 本人の確認なしに、複数AIが勝手に仕事を分担する

これらは目標から外しているのではなく、まだ届いていない部分です。

Locus Systemが目指しているのは、**すべての記憶が残り、AIが必要な瞬間に文脈として思い出し、利用者は裏側の保存方法を意識しなくてよい状態**です。

Markdownへの記録、LocusSave、RAG検索は、その目標へ向かう現在の実装です。

## まとめ

Locus Systemが目指す流れは、次のようなものです。

```text
AIと話す・作業する
  → すべての記憶が自動で残る
  → 必要な記憶をAIが自分で選ぶ
  → 過去の文脈を持ったまま会話や作業が続く
```

利用者は、保存、検索、ファイル名を意識しません。

現在は、その目標を次の仕組みで実現しています。

```text
必要な前提を読んで起動
  → AIと作業
  → Markdownへ記録
  → 検索DBとGitへ保存
  → 次のAIが同じ続きを読む
```

現在の記憶の正本はMarkdownです。

RAGは大きくなった記録から必要な部分を探す索引、APIは別の端末やAIから読む入口、LocusSaveは今回の変更を次回へ残す保存処理です。

将来、保存先や検索方法が変わっても、Locus Systemの目標は変わりません。すべての記憶を残し、必要な時に自然に思い出し、利用者が裏側を管理しなくてよいことが本体です。

この全体像を前提にすると、RAGのモデル変更、文章の区切り方、古い検索用データの削除といった改良が、Locus Systemのどの部分を触っているのか分かるようになります。

検索部分を実際に変更前後で測った記録は、次の記事にまとめています。
